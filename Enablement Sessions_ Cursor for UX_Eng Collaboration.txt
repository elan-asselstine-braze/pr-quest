What is a PR?
What a PR is
A pull request (PR) is a proposal for a change.
It is:
* A way to suggest a specific change
* A place for discussion and review
* Something engineers can modify, take over, or reject
A PR is not:
* A commitment that the change must ship
* A requirement that you maintain the code forever
A helpful mental model: a PR is a structured way to say: “Here’s a concrete suggestion, what do you think?”[a]

When it makes sense for designers to submit a PR
Designer-submitted PRs are most appropriate when the change is:
* Small and contained
* Easy to explain in one or two sentences
* Can be demonstrated with a Before and After screenshot
* Low-risk
Examples:
* Copy fixes[b]
* Documentation updates
* Token or variable usage fixes
* Minor visual tweaks
* Small component variants
* Comments or clarifications in code

When designers should not submit a PR
PRs are not appropriate for:
   * Architectural changes
   * Complex logic or data flow
   * Performance optimizations
   * Anything you don’t understand
Rule of thumb: If you can’t clearly explain what you changed and why, don’t submit a PR.


Live Demo (Follow-along)
   1. Open a repo in Cursor
   2. Create a new branch
   3. Make a very small change (for example):

      * Fixing copy
      * Updating a comment
      * Improving a README or doc

         4. Use Cursor to sanity-check:

            * “Explain what this file does.”
            * “Explain what I just changed.”
            * “Does this affect anything else?”

               5. Open a PR
               6. Write a simple PR description:
               * What changed
               * Why it changed
               * What did not change
               7. Spin up a PCE to preview your changes  in Github by commenting pce-it OR manually create PCE (possibly add how-to if this doesnt feel like too much detail)
               8. Attach image of the preview to the PR description
               9. Once automated tests are done (all check marks ✅), submit for review[c]
Call out during the demo:
               * Nothing breaks by opening a PR
               * Engineers still review everything
               * This is reversible and collaborative
